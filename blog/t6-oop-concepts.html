Classes vs Modules

Who's top dog?

July 26 2015

Divine right of objects

This week at DBC for our blog post, we were tasked with writing about a topic of our choice on some advanced concepts in Ruby. I chose to talk about the major differences between Ruby classes and modules.

Ruby classes and modules are objects in Ruby, (remember everything is an object in Ruby!) However they are in some respects more important that others since they tend to encompass so much of the functionality that comes with Ruby.

There is a hierarchy of objects in Ruby; in fact we can check the ancestors of any object with a simple method, #ancestors.

Example 1: We create a class and check the ancestry.

class Bicycle
end

#Since class is an object we can call Kernel methods on it
Bicycle.ancestors
#=>=> [Bicycle, Object, Kernel, BasicObject]

So here the real beauty of Ruby as a language is really starting to reveal itself. You can see a direct line of ancestry from the class that we created up to Object, then Kernel and finally, the building block of Ruby, BasicObject.

We can create our own hierarchy, let's say Gear and make it a child of Bicycle.

Example 2: creating our own hierarchy and inheritance path. 

class Bicycle
end

class Gear < Bicycle
end

Gear.ancestors
#=> [Gear, Bicycle, Object, Kernel, BasicObject]

Objects in the Gear class will now look first in Gear for methods when called, then into Bicycle and so on.

Now for Ruby modules, these are also objects (surprise surprise!) that contain methods and can be included in your Class definitions. It is worth noting that Kernel is a module and it contains your very basic objects such as Array, Integer, and basic methods as well such as puts.

Your objects will always look up into the hierarchy when a given method is called, and the first method that matches encountered will "win". Modules provide a handy way of inserting more functionality into your classes, and provide a way to keep 'best known' design practices such as modularity (hence the name) and single-purpose classes. I am still learning all about this, so I am sure that in the coming weeks it will become much clearer. For now it makes sense that you wish to make your classes as simple as possible.

So, returning to modules, they are handy to insert a method into a class to say, "override" one that is higher up in the chain of ancestry. This delves into a different but related topic of inheritance.

I am still only a humble Ruby beginner, but Ruby is for me quickly becoming a very nice learning experience and one which I can see why is popular. Thanks for reading.
